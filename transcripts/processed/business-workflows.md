## System Overview

The Hoodsly ecosystem consists of:

1. **Connector Platform 2.0** — Central WordPress plugin for managing product
   data, partner configurations, colors, SKUs, and attribute mappings
2. **HoodslyHub** — Order Management System that receives orders from all
   partner platforms
3. **WRHHub** — Fulfillment shop (Wood Range Hoods — standard products, Quick
   Ship)
4. **WiksHub** — Fulfillment shop (Wilkes — floating shelves, some hoods)
5. **Partner Sites** — WordPress, Shopify, or Magento storefronts that install
   the Hoodsly Partners Connector plugin

## Connector Platform 2.0 — Key Concepts

### Why Connector Platform Was Built

- Direct management from Hoodsly was not scalable for multiple partners
- Each partner needed custom: colors, pricing, SKU prefixes, product titles,
  FAQs
- Connector Platform centralizes all product/partner data
- Reduces dependency on ACF (Advanced Custom Fields) in CP 2.0

### ACF Field Groups

- Control: models, ventilation types, molding options, sizing modifications,
  order options
- Color options and conditional logic are tied to ACF data
- Changing ACF data has project-wide impact — must be done carefully
- CP 2.0 reduces ACF dependency

### Product Types

| Type                      | Description                                            | Routing       |
| ------------------------- | ------------------------------------------------------ | ------------- |
| Connector                 | Managed via Connector Platform, synced to partners     | Standard flow |
| Non-Connector             | Standalone products, imported but not platform-managed | Manual        |
| Floating Shelf ("Itself") | Visible on Floating Shelves tab; UPS courier           | WiksHub       |
| Quick Ship (QSP)          | One-In-One-Out inventory; fast fulfillment             | WRHHub        |

## Partner Onboarding — Step by Step

### Installation

1. Install "Hoodsly Partners Connector" WordPress plugin on partner site
2. Plugin appears under WP Admin menu

### API Key Generation

1. Admin generates API key in Connector Platform for the partner
2. Partner configures the API key in their plugin settings
3. API key determines: STG (staging) vs PRD (production) environment
4. Auto-detection: API key prefix indicates environment

### Partner Creation Page Fields

| Field                  | Required | Notes                              |
| ---------------------- | -------- | ---------------------------------- |
| Partner Name           | Yes      | Must match Hub Source Name EXACTLY |
| Partner Type           | Yes      | B2B or B2C                         |
| SKU Prefix             | Yes      | 3 uppercase letters, e.g. "TPB"    |
| Platform Type          | Yes      | wordpress / shopify / magento      |
| Website URL            | Yes      | Partner's storefront URL           |
| Color Style            | Yes      | select (B2B) or swatch (B2C)       |
| Ventilation            | Optional | Enable/disable for partner         |
| Trims                  | Optional | Enable/disable for partner         |
| Sizes                  | Optional | Enable/disable for partner         |
| Discount %             | Optional | Partner-specific discount          |
| Ventilation Discount % | Optional | Separate ventilation pricing       |
| API Key                | Auto     | Generated by platform              |

### Product Import

- Admin clicks "Import Products" on partner's Connector page
- Must accept: Terms of Service + MAP (Minimum Advertised Price) policy
- Products imported as connector products on partner site
- Difference: Connector products → managed by platform; Non-connector →
  standalone

### Color Management

- Global colors defined in Connector Platform
- Partner can override: display name and pricing per color
- Color style: Select dropdown (B2B) or Swatch visual (B2C)
- Catalog name displayed alongside color name for identification

## Order Flow — Complete Lifecycle

### Phase 1: Order Placement (Partner Site)

1. Customer browses products on partner WooCommerce site
2. Selects product attributes (size, ventilation, color, model)
3. Attribute mapping converts selections → Short SKUs
4. Customer fills checkout (requires valid US phone + email for BOL)
5. Order created in WooCommerce

### Phase 2: Order Hold

1. Order held for 24 hours (configurable per partner)
2. During hold: Customer can request cancellation
3. Admin/CSM can "Instant Release" to skip hold period
4. Orders containing customer notes → auto-routed to Manual Placement

### Phase 3: Order Forwarding to Hub

1. After hold period, order forwarded to HoodslyHub via API
2. HoodslyHub assigns unique ID (multi-item: baseId-1, baseId-2...)
3. Hub receives: Short SKUs, customer info, shipping details

### Phase 4: Fulfillment Routing

| Product Category | Destination | Courier    |
| ---------------- | ----------- | ---------- |
| Standard hoods   | WRHHub      | RL Carrier |
| Quick Ship (QSP) | WRHHub      | RL Carrier |
| Floating Shelves | WiksHub     | UPS        |
| Wood Hood        | WRHHub      | RL Carrier |

### Phase 5: BOL Generation

- RL Carrier requires valid US phone number and accurate email
- Shipper name must be correctly configured
- Failures: incorrect phone, invalid email, bad shipper name
- Fix: Edit customer details from Hub BOL editor → regenerates tracking

### Phase 6: Status Updates

Status flow: `Pending → Production → Finishing → Shipped`

Status sync path:

- WRHHub/WiksHub shop status change → HoodslyHub → Partner Site
- Requires: Partner Source Name in Hub Settings matches Connector Platform
  partner name exactly
- Caching delay: updated Source Names may not reflect immediately

### Phase 7: ERP & CSM Flow

- Orders also forwarded to ERP system
- CSM team manages: order statuses, estimated ship dates, delay notes
- "Post Meta Creator" plugin used for updating order meta directly from Hub
  backend
- Meta keys: `updated_est_ship_date`, `est_reason_note`
- Trigger sync: Change order status once after updating meta to push to partner
  site

## SKU Generation Logic

### How SKUs Are Built

```
[SKU Prefix][Master SKU]-[Size Code]-[Ventilation Code]-[Color Code]-[Model Code]
Example: TPB-WM-001-30-DUC-SS-STD
Where:
  TPB = Partner SKU prefix
  WM-001 = Master SKU for the product
  30 = Size: 30 inch
  DUC = Ventilation: Ducted
  SS = Color: Stainless Steel
  STD = Model: Standard
```

### Attribute Mapping

- Admin configures mapping: WooCommerce attribute value → Short Code
- Example: "stainless steel" → "SS", "ducted" → "DUC"
- Retrieved via API: `/wp-json/connector-platform/v1/attributes-mapping`

### Reduce Height Logic

- Some products support height reduction (e.g., custom fit)
- Admin configures reduction amount (e.g., 2 inches)
- SKU height code is adjusted accordingly

### Multi-Item Orders

- Each item in an order gets its own Hub order ID
- Format: baseOrderId-1, baseOrderId-2, etc.
- Single item orders: no suffix

## Damage Claims

### Process

1. Partner receives Hub access via dynamic role creation
2. Role is auto-created during partner onboarding in Hub Settings
3. Partner role restrictions: can only see their own completed orders
4. Partner views order → "File Damage Claim" button
5. Submits: damage type, description, images
6. Admin receives email notification
7. Admin reviews → sets claim type, confirms pickup week/date via email
8. Admin marks claim as Completed

### Access Control

- Main orders page: blocked for partner roles
- Other partner pages (Prime, KCD): blocked
- Partner can see: own completed orders + current order progress

## Critical Business Rules for Automation

1. **Partner Name Exact Match** — spaces vs underscores cause sync failures
2. **BOL Requirements** — US phone format, valid email, shipper name
3. **Order Hold** — orders wait 24h before forwarding (configurable)
4. **Customer Note Orders** — automatically go to Manual Placement
5. **Floating Shelf Courier** — UPS not RL
6. **QSP Inventory** — One-In-One-Out logic
7. **Status Trigger** — must change order status once to trigger meta sync
8. **Cache Delay** — Source Name updates may not reflect immediately
9. **Staging vs Production** — API key prefix determines environment
10. **Product Images** — loaded directly from Connector Platform URLs
